# Registry

> [!Abstract]
> Реестр соответствий документ → рантайм и тесты. Каждая строка фиксирует роль заметки, уровень авторитета, владельца-артефакта в коде или данных и шлюз, которым это проверяется. Реестр служит основой для синхронизации документации и как навигационный слой проекта.

___

## Определение

Registry – минимальная машинно-удобная карта между каноническими заметками CenterWay и их исполнительными артефактами. Он не пересказывает контракты, только связывает их с кодом и тестами. Формат – таблица; при переносе в JSON или YAML реестр остаётся стабильным по полям.

Поля каждой записи:

- `doc` – каноническая заметка в `ReOS/Projects/CenterWay/*`
- `role` – `project | canon | contract | spec | ops | decision | audit | resource`; составные роли допустимы через `+` (`spec+contract`, `ops+audit`, `ops+spec`)
- `authority` – `docs-first | code-first | data-first | mixed`
- `status` – `active | planned | partial | snapshot | superseded | archived`
- `source_of_truth` – repo-relative path/glob list через `;`
- `implemented_in` – repo-relative runtime/data path list через `;`
- `validated_by` – check list через `;` с префиксами `cmd:`, `manual:`, `planned:`
- `legacy_source` – provenance breadcrumb в `docs/legacy/**` или `docs/legacy/migration/**` как источник миграции

## Правила drift-gate

Прагматичный шлюз синхронизации канона и рантайма строится вокруг трёх обязательных блоков:

- **mandatory preflight** – агент читает обязательные точки входа в активный канон и доменный канон до начала работы;
- **canon compliance** – реализация не должна ломать действующие инварианты договорных заметок;
- **canon sync trigger** – заметные изменения поведения, токенов, CTA, логики маршрутов, контрактов данных или релизного гейта требуют обновления канона в том же цикле.

Минимум, который должен быть проверяемым:

- Frontmatter у активной заметки обязан содержать `role`, `authority`, `status`;
- `role` берётся из словаря `project | decision | canon | contract | spec | ops | audit | resource`; `snapshot` живёт только в `status`; составные роли допустимы только для `spec+contract`, `ops+audit`, `ops+spec`;
- `contract`, `spec`, `ops` и составные роли с этими компонентами обязаны иметь заполненные `implemented_in` и `validated_by`;
- `source_of_truth` и `implemented_in` должны быть machine-readable путями текущего репо или glob-паттернами над ними;
- Runnable-команды в `validated_by` должны разрешаться через `package.json` или существующий `scripts/**` путь;
- `docs/legacy/**` разрешён только как provenance reservoir, не как активный источник истины;
- Битые wikilinks внутри `ReOS/Projects/CenterWay/**` не допускаются.

## Применение и связи

Реестр – точка входа для достоверной навигации: читаешь его как таблицу, находишь тему, уходишь в канон или прямо в рантайм или тест. Он же – основа для автоматического шлюза drift-gate и для будущего подъёма канонов в `BridgeOS` после стабилизации модели авторитета.

Удобная для человека навигация по канону – в [[CenterWay]]. Принципы управления – в [[Мета-аудит]] (раздел о шлюзе).

## Дополнение к реестру рантайма

Runtime manifests in the CenterWay repository are execution serialization of RAverse, not independent canon.

Текущий смысловой слой рантайма:

- Data/design-tokens/cw.tokens.json: примитивы, смысловые алиасы, переопределения режимов, рецепты компонентов, наложения маршрутов.
- Data/generator/route_family_contracts.json: границы семейств маршрутов, допустимый порядок блоков, обязательные смысловые роли, разрешённые рендереры.
- Data/generator/block_manifests.json: смысловую роль блока, семейство, вопрос пользователя, границу маршрута, рендерер, рецепты токенов и варианты раскладки.
- Data/generator/screen_manifests.json: сборку маршрутов и стек блоков для каждой публичной поверхности.

Проверочные шлюзы: npm run canon:guard, npm run generator:validate, npm run semantic:audit, npm run ds:qa.
