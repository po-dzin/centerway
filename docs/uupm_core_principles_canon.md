# UUPM Core Principles Canon for CenterWay

Источник: `nextlevelbuilder/ui-ux-pro-max-skill` (main, проверено 2026-03-30).  
Роль документа: зафиксировать базовые принципы UUPM в формате, совместимом с архитектурными и дизайн-доками CenterWay.

## 1) Принципы, которые принимаем

1. `System-first`, не `screen-first`.
UI строится через повторяемые правила (паттерн -> стиль -> токены -> компоненты), а не через разовые макеты.

2. Доменно-ориентированный выбор решений.
Структура экрана и визуальный стиль подбираются от типа продукта и сценария, а не “по вкусу”.

3. Явная связка:
`layout pattern + typography + color logic + interaction effects`.
Эти части нельзя выбирать изолированно.

4. Антипаттерны фиксируются заранее.
Перед реализацией указывается, что запрещено для конкретного сценария (например, hype-стиль, визуальный шум, низкая читабельность).

5. Accessibility и usability как базовый контракт.
Минимум: видимый focus, достаточный контраст, поддержка `prefers-reduced-motion`, понятные hover/active состояния.

6. Адаптивность по контрольным точкам.
Проверка на опорных ширинах: `375 / 768 / 1024 / 1440`.

7. Token-driven реализация.
Цвета/отступы/радиусы/типографика применяются через токены и семантические алиасы, а не через “сырые” значения в компонентах.

8. Pre-delivery quality gate.
Перед релизом обязательный короткий чеклист качества (визуальная консистентность, состояния, контраст, адаптив, интерактивность).

## 2) Адаптация к CenterWay (обязательные ограничения)

1. Brand/claim политика CenterWay выше UUPM.
Любой UUPM-паттерн проходит фильтр `centerway_*_passport`, claim matrix и semio-доков.

2. Токены только через `--cw-*`.
Любые palette-идеи UUPM проходят маппинг в существующий token-canon.

3. “Calm clarity” вместо агрессивной конверсии.
Отклоняем визуальные решения, которые перегружают внимание или ломают trust-first подачу.

4. Никаких шаблонов “как есть”.
Берем принципы и эвристики, но не копируем style bundles без адаптации под контекст CenterWay.

## 3) Применение по контурам проекта

### 3.1 Landing/Marketing
- Использовать UUPM для: секционной иерархии, CTA-ритма, accessibility чеков.
- Не использовать UUPM как источник final copy и claim-формулировок.

### 3.2 Admin Panel
- Использовать UUPM для: иерархии таблиц/дашбордов, плотности информации, состояний loading/empty/error.
- Фиксировать UI-инварианты: стабильные табы, одинаковая геометрия состояний, отсутствие layout-shift.

### 3.3 Design System
- Поддерживать слой токенов: primitive -> semantic -> component.
- Перед добавлением нового компонента проверять совместимость с существующими admin/landing паттернами.

## 4) Операционный процесс внедрения

1. Определить экран/модуль и его продуктовый контекст.
2. Выбрать 1-2 UUPM-подхода (структура + визуальная логика).
3. Прогнать через ограничения CenterWay (бренд, claims, токены, trust-tone).
4. Реализовать через существующие компоненты и `--cw-*` токены.
5. Пройти pre-delivery gate (accessibility, responsive, states, consistency).
6. Зафиксировать изменения в паспортах/дизайн-доках при изменении контрактов.

## 5) Быстрый чеклист перед merge

- [ ] Нет сырых цветовых/типографических значений вне токенов.
- [ ] Focus/hover/active состояния читаемы и консистентны.
- [ ] Нет сдвига геометрии при переключении состояния (tabs/buttons/badges).
- [ ] Нет конфликтов с claim/policy документами.
- [ ] Проверены ключевые брейкпоинты (`375/768/1024/1440`).
- [ ] Подтверждено, что UI не уходит в шаблонный “generic wellness” стиль.
